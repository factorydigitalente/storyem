# 命令
以下の物語の概要をもとに、CSV出力ルールに従ってCSV形式でルールCSVとシナリオCSVを出力してください。
解説や前置きは不要です。まずルールCSVを出力し、空行を1行入れてからシナリオCSVを出力してください。それ以外の文章は一切出力しないこと。

★重要：出力前の自己検閲ルール
CSVを出力する前に、以下の3点を必ず内部でセルフチェックしてください。以下の検証を完了し、論理的な矛盾がないことを確認するまで、決して出力してはいけない。

1. 到達可能性: id:1 から全ての id へ、選択肢を辿って到達可能か？（孤立したIDはないか）
2. フラグの整合性: effect で入手したアイテムやゴールドを、その後の options の条件判定 [condition] で適切に使用・消費しているか？
3. 行き止まりの排除: 条件付き選択肢しかないパラグラフにおいて、条件を満たさないプレイヤーが「選択肢ゼロ」で詰む箇所はないか？
4. 入手手段の確定（最重要）: options の条件判定 [condition] で要求しているアイテムやゴールドがある場合、それ以前のパラグラフの effect で、プレイヤーが確実にそれを入手できるルートが物理的に存在するか？（「探索する」という選択肢があるのに、その移動先で item+ を書き忘れるミスを厳禁とする）
5. リソースの循環確認:ショップに必須アイテムがある場合、その購入費用を稼ぐための手段（戦闘や探索）の報酬には必ず ! を付け、リピート可能（例: !gold+50）にすること。一度きりの報酬のみで必要金額に達せず、物理的に進行不能になる構成を厳禁とする。

生成するCSVは、まず内部で構造チェックを行い、不備があればその場で修正すること。
すべてのチェックをパスしたことを前提として、完成したデータのみを出力せよ。

・数値は半角数字
・カンマ区切り
・1行目は必ず項目名

・引用符（""）の徹底（最重要）: text, options, effect の3つの列は、必ず列全体を半角ダブルクォーテーション "" で囲ってください。 > 特に、複数の選択肢や複数の効果をカンマで区切って記述する場合、囲わないと列がズレてしまい、データが正しく読み込まれません。
　正しい例: 1,normal,"文章","選択肢A→2,選択肢B→3",,,,,bgm.mp3,img.jpg,"item+key,skill+1"

8: 茶（土・遺跡・レトロ）
9: 濃茶（泥・腐敗・絶望）

▼options（選択肢）の基本の書き方
形式：　表示テキスト→移動先ID
複数ある場合：　左へ→2,右へ→3

注意：
・「→」は全角矢印
・カンマで区切る
・移動先は必ず存在するID

▼options（選択肢）の「フラグ・条件チェック」
特定のアイテムやフラグを持っていないと選べない選択肢を作れます。
形式： [アイテムIDまたはフラグ名]表示テキスト→移動先ID
1. 所持金チェック： [gold:200] （200G以上必要）
2. アイテム所持： [item_id] （そのアイテムが必要）
3. 未所持チェック： [!item_id] （そのアイテムを持っていない時だけ出現）
例： [gold:300]鉄の剣を買う→21
例： [silver_key]カギを使って扉を開ける→5

▼text の書き方
改行はしないこと。

▼戦闘の仕組み
戦闘は、2d6 + 戦技　で判定されます。
勝利 → 敵に2ダメージ
敗北 → 自分に2ダメージ
幸運使用で増減
幸運は使うたび1減少

▼effect（特殊効果）
特定のパラグラフに自動効果を付けられます。
効果によってHPが0になった場合は、その時点でゲームオーバーとなります。

ステータス増減の形式（通常は初回訪問時のみ）：
hp+3
hp-2
skill+1
luck-1
food+1

アイテム入手　例）item+アイテムID（例: item+silver_key）

ゴールド操作　例： gold+100, gold-50（例: gold+500）

※先頭に「!」を付けると、訪れるたびに繰り返し効果が発動します（例: !hp-2）。
※複数を組み合わせる場合はカンマで区切ります（例: gold-300,item+iron_sword,skill+3）。
※アイテムとゴールドを組み合わせるとショップにできます。（例：gold-300,item+herb）

例：以下の場合、同じidを訪れるたびにhp-2となります。
11,normal,"罠の道だ！","戻る→1",,,,,,,!hp-2
例：以下の場合、同じidを訪れても、2回目以降はfoodは増えません。
31,normal,"干し肉を見つけた。","進む→4",,,,,,,food+1

食事ポイントとして「eat」を書くと、食料がある場合にHPを4回復できるボタンが表示されます。
「eat」は1度きり、「!eat」は食料がある限り何度でも使用可能です。
例：
4,normal,"ここは安全そうだ。食料を好きなだけ食べても構わない。","進む→5",,,,,,,!eat

bgm_fight.mp3

▼背景画像（bgi）
そのパラグラフの専用背景
空欄ならルールCSVで指定した背景を使用

★よくある記述ミス
・ルールCSV、シナリオCSVのヘッダー行を漏らさないこと ・カンマ不足：列数が足りないと正常動作しません。すべての行は必ず列数が同じ（カンマの個数が同じ）であること（末尾が空でもカンマを打つ）。
・IDの重複：同じIDは使わないこと
・存在しないIDへ移動しないこと
・options内の矢印は必ず「→」を使う。「->」などは禁止。
・戦闘行の整合性: battleタイプでは、5列目(敵名)から8列目(勝利後ID)までを必ず埋めること。
・引用符の管理: textとoptionsとeffectは必ず半角のダブルクォーテーション "" で囲むこと。また、テキスト内での二重引用符（"）の使用は厳禁。セリフや引用は必ず一重引用符（'）または「」に置換すること。

・アイテム入手と条件分岐の分離: 同一ID内で「アイテム入手(effect)」と「そのアイテムを条件とする選択肢(options)」を同時に記述しないでください。入手と使用はパラグラフを分けてください。
　悪い例：6,normal,"鍵を得た","[key]扉へ→7",,,,,,,item+key（ボタンが反応しない恐れあり）
　良い例：6,normal,"鍵を得た","次へ→61",,,,,,,item+key
　　　　　61,normal,"鍵を持っている。","[key]扉へ→7",,,,,,,

・進行不能の回避（デッドロック禁止）: 選択肢に条件[condition]を付ける場合、その条件を満たしていないプレイヤーが移動先で「選べる選択肢がゼロ」にならないよう、必ず「条件なしの選択肢」を混ぜるか、移動前の時点で条件チェックを行ってください。
　悪い例：2,normal,"城へ着いた","次へ→3",,,,,,,
　　　　　3,normal,"扉がある","[key]開ける→4",,,,,,,（鍵がないと詰む）
　良い例：2,normal,"城へ着いた","[key]扉へ→3,鍵を探す→5",,,,,,,（移動前にチェック）
　または：3,normal,"扉がある","[key]開ける→4,戻る→2",,,,,,,（戻る選択肢がある）

✨ 生成後の便利な追加命令 (必要に応じて、AIに続けて入力してください)

🛠️ 「全行の構文エラー、バトル項目の未入力、および全選択肢からゴールへの到達可否を、一項目ずつデバッグして修正して。到達できないパラグラフがないか精査して。CSVのヘッダー行も忘れないで。」
アイテムフラグを使用した場合など、AIは1回の生成では整合性のあるデータを生成できない場合があります。その場合はこの命令を実行することで、徹底的な修正をし、精度の高いデータを再生成してくれやすくなります。
⏩ 「続きを書いて」「パラグラフ[番号]から再開して」「最後まで出力して」
AIの制限により物語が途切れることがあります。その場合はこの命令を実行することで、続きを出力してくれます。
